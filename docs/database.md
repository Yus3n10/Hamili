# Database Schema

All tables use an auto-incrementing integer primary key `id`. Foreign keys to `users.id` cascade on delete, so removing a user cleanly removes their data.

## `users`
| Column | Type | Notes |
|---|---|---|
| id | PK | |

| password_hash | varchar(255) | bcrypt |
| preferred_name | varchar(100) | used for AI personalization |

| monthly_salary | numeric(12,2) | nullable |
| allowance | numeric(12,2) | nullable |
| financial_goal_text | varchar(500) | nullable, free text |
| created_at | timestamptz | |

## `budgets`
| Column | Type | Notes |
|---|---|---|
| id | PK | |
| user_id | FK → users | indexed |

## Migrations

Schema changes always go through Alembic — never edit the database by hand:

```bash
# After changing a model in app/models/
alembic revision --autogenerate -m "add loan_balance to recurring_items"
alembic upgrade head
```

Always review autogenerated migrations before running them; Alembic can miss things like column renames (it sees them as drop+add).
